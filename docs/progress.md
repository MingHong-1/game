### 阶段 2D.1 战斗属性与伤害结算内核

- 新增无 Phaser 依赖的 `src/battle/combat/`：统一战斗属性、确定性修改器、伤害类型/标签、星级倍率、穿透减伤、伤害请求/结果、纯计算解析器和幂等应用账本。
- `CombatStats` 区分 Definition 输入与运行时不可变快照；默认值保持原型可运行，暴击率限制 0～80%、暴击伤害限制 100%～300%、穿透比例限制 0～100%，非法非有限值、负值和非正攻速明确失败。
- 修改器采用基础值 → add → multiply → override → 合法范围；override 按高 priority、稳定 sourceId 决定，聚合提供逐属性调试明细且不依赖输入顺序。
- 当前单体基础攻击继续使用原职业 `pathProgress` 索敌、攻击间隔和追踪弹道；命中时统一结算，伤害事件先于一次性死亡事件，抵达和死亡互斥语义保持不变。

- 疾风猎手从英雄定义获得 10% 基础暴击率，其他当前英雄为 0；默认暴击伤害 180%。合法判定只消费一次独立 `combatRng`，非法/死亡目标、禁暴击、DoT 和 0% 暴击不消费。
- 物理读取护甲，火/冰/雷/自然读取抗性，真实伤害绕过防御；百分比穿透先于固定穿透，减伤公式为 `defense/(defense+100)` 且上限 70%，承伤倍率位于防御后。
- `damage-applied` 事件携带完整、不可变的 `DamageResult`；表现桥只翻译为命中表现与安全音频事件，不能反向扣血。`?combatDebug=1` 可在开发构建输出每次成功应用伤害的紧凑日志。
- 重置时清空 request 序号、已应用账本、弹道、事件并重建同 seed 的 `combatRng`；同种子、同输入恢复相同伤害和暴击序列。

### 阶段 2C.4a 怪物视觉回退修复

- 分层诊断结果：固定步长模拟进度单调，插值后的 `renderPathProgress` 也单调；问题发生在显示世界坐标映射层。
- 根因是 `PathGeometry.sampleAt` 在折线节点处直接从上一段法线切换到下一段法线。中心线位置连续，但外侧 lane 的法线偏移不连续；原型路径采样中最大单次跳跃约 185 像素，表现为停顿、向后跳再继续。
- `PathGeometry` 现在为每个顶点计算相邻单位切线的归一化平均，并在线段内部连续插值切线/法线；配置节点、中心线长度、laneOffset、jitter 和权威进度不变。
- `BattleSimulation` 在开发模式断言同一实例的下一模拟进度不得低于当前进度；未加入 `max(lastRenderedProgress, renderedProgress)` 之类掩盖性钳制。
- 新增 `EnemyProgressRegressionMonitor`，按稳定 enemy ID 区分模拟回退和显示进度回退，记录 previous/current、alpha、renderedProgress/X/Y、帧 delta、accumulator、步数、丢帧、倍速和状态；同一实例同类回退只输出一次 `[EnemyProgressRegression]`。
- 丢弃追赶时间后显式同步 previous/current；visibility、失焦、重置和新局边界继续 snap，避免重新使用旧 previous 状态。
- `StableViewRegistry` 按实例 ID 绑定敌人和弹道视图，数组重排、删除和插入不改变其他视图身份；删除时清理 Tween 和显示对象。
- 敌人结构改为 `rootContainer → visualContainer → 图形/生命条`。运动根节点只有 `writeEnemyMotionPosition` 写入位置，受击透明度和缩放 Tween 只作用视觉子节点。
- 没有调整敌人速度、`traversalTimeSeconds`、波次、伤害、能量、召唤、格位、职业阈值或随机流。

### 阶段 2C.4 固定步长渲染插值

- 根因确认：渲染 FPS 可超过 100，但显示层此前只读取每 20 毫秒更新一次的当前模拟坐标，同一坐标会连续展示多个渲染帧，形成 50Hz 阶梯；长帧或后台恢复产生的大 delta 又会在一次画面刷新前执行多步模拟，表现为短距离跳跃。
- 新增 `RenderInterpolation` 纯函数，统一限制 `alpha = accumulator / fixedStep` 到 0～1，并提供无副作用数值插值。
- 敌人运行时保存 `previousPathProgress/current pathProgress`；快照先插值进度，再通过 `LanePathGeometry` 映射 `renderX/renderY`，避免跨曲线路点切角。lane、jitter 和权威 `pathProgress` 均未改变。
- 追踪弹道保存 `previousX/Y` 与当前坐标，并只在快照中生成插值显示坐标；弹道命中仍由固定步骤决定。
- 新生成对象的 previous/current 初始化一致；暂停、重置和状态终止会同步前后值；死亡或抵达对象立即从快照移除，不会因插值再出现一帧。
- 每帧最多执行 5 个固定步骤，累积器最多保留 100 毫秒模拟时间；输入帧继续受 250 毫秒上限保护，超出的时间丢弃并累计到 `droppedSimulationTimeMs`，避免无限追帧和页面冻结。
- 页面隐藏或窗口失焦时停止接收真实时间并清空插值余量；恢复后忽略首个旧 delta、重新同步时钟，不追赶整个后台停留时间。
- 开发调试面板新增最近/峰值帧耗时、模拟步数、alpha、丢弃时间、33/50/100 毫秒长帧计数、恢复同步次数、敌人/弹道/Tween 数。
- `BattlefieldView` 移除敌人和弹道同步中每帧创建的活动 ID `Set`，改用递增帧标记复用现有视图；静态背景和道路仍只创建一次，删除视图时继续清理 Tween。未引入对象池、ECS 或大规模性能重构。
- Canvas 策略经检查保持不变：逻辑缓冲仍为 1280×720，未额外乘以 `devicePixelRatio`，页面容器上限 1600×900，5K 显示不会产生无上限后备缓冲。
- `system-models-v1.md` 只同步“渲染同步已实现”的代码事实，没有修改 `traversalTimeSeconds`、通道、职业阈值、随机、召唤、能量或格位模型。

- 战斗区域调整为 `y=66～500`，中心路径从 `(640,72)` 延伸至 `(640,464)`，通过 9 个采样点形成两次柔和横向方向转换，有效曲线长度超过 500 像素。

### 阶段 2C.2a 波次随机子流解耦

- 新增 `WaveRandomStreams`，直接从当前 run seed 按关卡 id、波次 id 和用途派生每波独立流。
- 派生格式为 `${runSeed}:level:${levelId}:wave:${waveId}:composition|lanes|jitter`。
- `waveCompositionRng` 只抽取敌人池；`waveLaneRng` 只由 `LaneSpawnPlanner` 消费；`waveVisualJitterRng` 只生成局部视觉错位。
- `BattleSimulation.reset` 为每个波次重建三条流；同种子重演恢复相同组成、通道与 jitter。
- 波次日程校验改用无随机状态的通道候选纯函数，不在构建日程时推进任何流。
- 改变 jitter 消费或通道消费不再改变敌人组成；改变前一波敌人数不再改变后一波组成、通道或 jitter。

- `LanePathGeometry` 从中心路径、走廊宽度和路径法线派生 3～7 条通道，不复制多套路点。

### 阶段 2C.1 格位与正式随机局

- `battleConfig` 使用 0/5/10/15/20/25 次成功召唤对应 5/6/7/8/9/10 格，25 次后不再扩格；费用档位保持不变。

- 每轮英雄袋包含 4 名英雄各 2 份，共 8 张。
- 使用 `SeededRandom.shuffle` 执行 Fisher–Yates 洗牌，逐张抽取；抽完后生成并洗牌下一袋。
- 波次按组成、通道、jitter 三种每波子流隔离，召唤使用 `summonRng`；召唤操作不会改变波次随机序列。

### 满格拒绝契约

`SummonSystem.attemptSummon` 先以“下一次成功召唤次数”预检格位容量。

当前为第 9、14、19、24 次成功召唤且已满格时，只要能量足够，系统在同一同步事务中：

1. 确认下一次召唤会扩格。
2. 先解锁第 7、8、9、10 格之一。
3. 扣除当前档位能量。
4. 从英雄袋抽取一个结果。
5. 将英雄放入刚解锁的新格位。
6. 提交成功召唤次数。

其他满格时：

- 召唤按钮禁用。
- UI 显示“请先合成或重构英雄”。
- 能量、成功召唤次数、费用档位、解锁格位和扩格进度保持不变。
- 英雄袋不抽取、不预取、不丢弃结果。
- 不向战斗模拟添加英雄，也没有召唤成功事件或后续效果。
- 腾出已解锁格位后，下一次成功召唤与没有发起满格请求的同种子对照结果一致。

### 阶段边界

- 扩格召唤例外覆盖第 10、15、20、25 次的满格边界，不需要临时弃置规则。
- 任一扩格之后若再次满格且下一次不会扩格，仍必须先腾出格位。
- 阶段 3 需要实现始终可用的“遣散英雄”作为最终防卡死安全阀；本阶段只记录需求，不提前实现合成、遣散规则或 UI。

# 《星核守望》核心游戏设计基线

## 产品定位

《星核守望》是一款原创的、单机、无氪金的随机合成塔防游戏。每局目标时长为 3～5 分钟，核心体验来自有限格位和共享能量下的短局决策，而不是长期付费养成。

明确不包含：抽卡付费、体力、广告、英雄碎片或其他强制商业化系统。

## 已确定的单局规则

### 阵容与格位

- 每局开始前选择 4 名英雄，构成本局的英雄池。
- 初始拥有 5 个英雄格位。
- 每完成 5 次召唤，扩展 1 个英雄格位。
- 每局最多拥有 10 个英雄格位；累计 25 次成功召唤后达到上限。
- 普通满格召唤零副作用拒绝；若下一次成功召唤会扩格，则先扩格并原子完成召唤。

### 能量、召唤与技能

- 消耗能量，从本局 4 名英雄中随机召唤 1 名英雄。
- 英雄召唤和技能购买使用同一种能量。
- 玩家需要在立即增加场上单位、购买技能和保留能量之间做选择。
- 正式经济采用十位数能量：初始 150，召唤费用按 50/60/70/80/90 分档，三组技能基线为 80/130/180，后续刷新 30。
- 正式能量来源使用自然恢复、波次预算、清波奖励、技能和遗物；怪物通过 `energyWeight` 瓜分波次预算，不再按固定击杀值决定总能量。
- 上述经济架构已冻结，代码将在阶段 2E 迁移；具体总量和节奏仍需第一关验证。当前 15 初始、5～9 召唤和固定击杀奖励只属于原型实现。

### 常规合成

- 只有“相同英雄且相同星级”的两个单位可以进行常规合成。
- 合成消耗这两个单位。
- 合成结果从本局 4 名英雄中随机生成，不保证保留原英雄。
- 结果英雄的星级比参与合成的星级高 1 星。
- 最高 4 星；4 星为满星，不得继续升星合成。1→2、2→3、3→4 的保留素材英雄概率和其他英雄分配以 `system-models-v1.md` v1.1 为准。

这套规则有意制造阵容不确定性：合成提升战力和释放格位，但可能改变英雄类型，因此玩家需要权衡阵容结构、格位压力与随机结果。

### 每局限次能力

- **紧急重构：** 每局可以使用 1 次。
- **命运校准：** 每局可以使用 1 次。

命运校准可以令指定目标英雄在本次结果中达到 100%；紧急重构允许两名不同但同星英雄进行随机重构。两项能力均每局一次，第二次必须拒绝并给出 UI 反馈。原子提交、保护阈值和交互细节在阶段 3 按 `system-models-v1.md` 落地，不能在代码中隐藏新决定。

## 单局体验目标

一局应围绕以下循环展开：

1. 观察敌人、当前阵容、格位和能量。
2. 在召唤英雄、购买技能或暂缓消费之间选择。
3. 判断是否进行合法合成，以换取星级提升和格位空间。
4. 必要时使用每局一次的紧急重构或命运校准修正局势。
5. 应对逐步加压的波次，在 3～5 分钟内达成胜利或失败结果。

随机性应产生可管理的风险，而不是完全替代决策。界面需要在行动前清楚显示费用、格位、可合成目标、限次能力状态和关键概率规则。

## 胜负与节奏

已确定：单局目标时长为 3～5 分钟。

当前已经确定并实现：敌人抵达后伤害星核，星核生命为 0 失败；Boss 死亡胜利；固定步长、暂停、1×/2×、宽走廊、职业攻击区域和自动攻击均已建立。

仍需第一关验证：完整波次数量和间隔、3～5 分钟时长、十位数能量增长曲线、英雄/敌人/技能数值、Boss 机制和失败原因可读性。先验证循环，再扩充内容。

## 战场走廊与通道

- 标准战场使用从上向下、逐渐向星核收束的宽走廊，而不是只能容纳一个敌人的细路。
- 所有敌人共享一条中心路径和统一的 `pathProgress`；默认使用 5 条隐形横向通道，关卡可配置为 3～7 条。
- 通道只派生敌人的世界坐标，不改变职业攻击阈值、抵达判定或最接近星核的索敌语义。
- 世界坐标可供后续圆形、扇形、穿透和区域能力查询；通道不是技能命中的隔离区。
- 波次可选择随机、均匀、中央突袭、两翼、全线怪潮和 Boss 中央/小怪两翼队形。
- 每波的敌人组成、通道与局部错位必须使用相互独立的可复现随机子流，不能使用 `Math.random()`。
- 波次子流按 `run seed + 关卡 id + 波次 id + 用途` 派生；前一波数量、通道抽样次数或纯视觉 jitter 算法变化不得串扰后续波次和其他用途。

标准英雄指挥台为底部居中的两排五格。格位继续只承担容纳和展示，不提供站位射程或属性加成。

## 敌人推进与波次节奏

- 敌人的权威移动参数是 `traversalTimeSeconds`，表示 1 倍速下从 `pathProgress=0` 到 `1` 的秒数；道路像素长度只参与世界坐标映射。
- 当前原型基线为：快速敌人 14 秒、普通敌人 22 秒、重甲 30 秒、精英 28 秒、Boss 45 秒。
- 通道、laneOffset 和视觉 jitter 不改变推进时间；固定步长、暂停、1×/2× 继续由战斗时钟统一控制。
- 运行时保留 `movementRate` 接口供阶段 2D 减速或加速使用，本阶段固定为 1，不实现状态效果。
- 每关通过 `initialPreparationSeconds` 配置准备期，原型为 8 秒；准备期间可以召唤，但敌人和第一波尚未生成。
- `wavePreviewSeconds` 配置波次预告窗口，原型为 5 秒；预告包含波次编号、主要敌人、威胁类型和通道队形。
- 波次使用相对正式战斗开始的时间轴并允许重叠，不要求上一波清空。当前三波原型用于验证长推进与重叠；完整 3～5 分钟内容仍需第一关垂直切片验证。
- 22 秒普通敌人在射手、法师、辅助、战士阈值后的理论攻击窗口分别约为 20.9、17.6、14.3、6.6 秒；14 秒快速敌人的战士窗口约为 4.2 秒。

## 战斗属性与伤害结算

- 当前单体基础攻击统一在弹道实际命中时提交伤害请求；目标选择、攻击间隔和弹道速度仍沿用原固定步长规则，表现动画不决定命中或伤害。
- 伤害类型为物理、火焰、冰霜、雷电、自然和真实；未配置的基础攻击安全默认为物理。物理读取护甲，四种元素读取抗性，真实伤害绕过防御。
- 基础攻击使用 `basicAttack` 与 `projectile` 标签；skill、dot、summon、aoe 仅建立类型边界，尚未实现对应玩法。
- 基础攻击伤害顺序为：基础值 → 1～4 星倍率 → 通用/标签/Boss 倍率 → 暴击 → 防御减伤 → 目标承伤倍率 → 实际与溢出伤害。内部不做中间整数取整。
- 星级伤害倍率暂为 `1.00/1.65/2.65/4.10`；疾风猎手基础暴击率 10%，其他当前英雄 0%，默认暴击伤害 180%，暴击率/暴击伤害上限为 80%/300%。这些数值均需第一关验证。
- 护甲和抗性减伤暂为 `defense/(defense+100)`，上限 70%；有效防御为 `max(0, defense×(1-percentPenetration)-flatPenetration)`。当前敌人防御默认 0，正式分配留给 2D.4。
- 暴击只使用 `combatRng`。非法请求、死亡目标、禁暴击、DoT 和 0% 暴击不消费；合法判定固定消费一次，保证同种子与同事件顺序可复现。
- 属性修改器统一采用基础值 → add → multiply → override → 合法范围，override 使用 priority 与稳定 sourceId 决定；装备、技能和状态尚未成为实际修改器来源。

## 内容范围

- 最终目标：10 章，共 100 关。
- 当前范围：只开发可验证核心循环的原型和第一关。
- 第一关稳定后，再用配置方式扩展第一章 10 关。
- 在核心循环、时长和难度曲线验证前，不批量生产 100 关，也不加入完整装备系统。

## 数据与可测试性原则

- 英雄、敌人、技能和关卡数据配置化，规则逻辑与具体数值分离。
- 关卡配置表达敌人组合、生成节奏和参数修正，不复制整套战斗代码。
- 每局使用 run seed，并为波次、召唤、战斗、合成和技能隔离随机流；波次内部继续拆分组成、通道和视觉 jitter 三种每波独立子流。
- URL seed 用于调试和分享；普通新局生成新 seed；同种子重演恢复相同随机序列。
- 核心规则不依赖 DOM 或具体渲染框架，便于单元测试。
- 战斗状态以单一可信来源为准，UI 只展示状态并提交玩家意图。
- 存档只用于局外进度和设置；单局重开、断点续局是否支持需要另行决定。

## 核心原型验收方向

核心原型至少需要证明：

- 4 名英雄阵容能实际影响随机结果。
- 召唤与技能的共享能量能形成真实取舍。
- 5 至 10 格的扩展节奏既带来压力，也不会频繁造成无解局面。
- 合成的随机换英雄规则可理解、可复现问题且有足够反馈。
- 两项每局一次能力能缓解极端随机结果，但不会抹掉所有风险。
- 普通玩家能在 3～5 分钟内完成一局，并理解胜负原因。

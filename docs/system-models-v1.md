## 文档定位

本文档是后续开发的统一系统规则来源。它同时记录“最终设计模型”和“当前代码事实”，二者不一致时必须明确标为待迁移，不能把已经确定的设计误写成已经实现，也不能因旧文档或原型代码仍存在而恢复旧规则。

优先级如下：

1. 本文档中的已冻结模型定义后续目标。
2. `docs/current-state.md` 记录当前实现事实。
3. `docs/progress.md` 记录历史完成项和验证证据。
4. 源码与测试决定“当前究竟实现了什么”，但不自动推翻本文档中的后续冻结模型。
5. 旧阶段文档只保留历史意义；与本文档冲突时不得作为新实现依据。

## 状态词汇

| 状态 | 含义 |
| --- | --- |
| 已冻结 | 大方向、职责边界和核心接口不再随意变化 |
| 已实现 | 当前源码已经存在，并有相应测试或工程验证 |
| 待实现 | 设计已经确定，但当前代码尚未完成 |
| 待首关验证 | 需要第一关完整垂直切片验证体验、时长、概率或数值 |
| 可配置调整 | 可以只改配置或数值，不改变模型含义和架构职责 |

“架构状态”和“数值状态”可以不同。例如敌人推进架构已冻结且已实现，但 14～45 秒仍属于待首关验证、可配置调整。

## 1. 单局运行与时间模型

| 模型名称 | 设计状态 | 代码状态 | 当前源码位置 | 当前测试位置 | 下一实现阶段 | 需要首关验证 |
| --- | --- | --- | --- | --- | --- | --- |
| 单局状态机、固定步长与渲染同步 | 架构已冻结；时间数值可配置调整 | 已实现 | `src/battle/BattleState.ts`、`BattleStateMachine.ts`、`BattleSimulation.ts`、`RenderInterpolation.ts`、`EnemyProgressRegressionMonitor.ts`、`src/data/levelDefinitions.ts`、`src/ui/BattlefieldView.ts` | `BattleStateMachine.test.ts`、`BattleSimulation.test.ts`、`EnemyTraversalTiming.test.ts`、`RenderInterpolation.test.ts`、`EnemyProgressRegression.test.ts` | 阶段 2D 扩展状态效果；阶段 4 完整波次 | 是 |

正式规则：

- 单一状态机包含准备、战斗中、暂停、技能选择、胜利、失败。
- 开局 8 秒准备倒计时不新增第二套状态机；当前通过 `Running` 内的关卡时间轴表达。
- 战斗模拟使用 20ms 固定步长；玩法状态只在固定步骤更新，渲染帧率不得改变模拟结果。
- 连续移动对象保存前一与当前模拟状态；显示层以 `alpha = accumulator / fixedStep` 在两者之间插值。敌人以插值后的 `pathProgress` 重新映射宽走廊世界坐标，插值结果不参与索敌、伤害、抵达或胜负。
- 当前长帧策略为每帧最多 5 个模拟步骤、最多保留 100ms 累积模拟时间；输入 delta 仍以 250ms 为上限，超出可追赶范围的时间会丢弃并进入调试计数，不允许无限追赶。
- 页面隐藏或窗口失焦时暂停接收真实时间并清空插值余量；重新可见或聚焦后忽略首个旧 delta 以重同步时钟，不追赶后台时间。
- 当前不存在击退能力，同一敌人实例的模拟和渲染 `pathProgress` 必须单调不减；开发模式以稳定实例 ID 检测并一次性输出结构化回退诊断，不通过单调钳制掩盖错误。
- 丢弃追赶时间、visibility 重同步、新生成和重置边界必须把 previous/current 同步到同一权威状态；敌人运动根节点只能由插值显示坐标写入，受击 Tween 只能作用视觉子节点。
- 支持 1×和 2×；暂停和技能选择期间模拟时间停止。
- `initialPreparationSeconds` 默认 8 秒；准备期间可以召唤，敌人不生成，不新增自然能量。
- `wavePreviewSeconds` 默认 5 秒；预告不控制真实生成，生成日程是权威来源。
- 波次可以重叠，不要求上一波清空。
- UI 动画、Tween、按钮颜色和提示回调均不能决定真实状态、扣费、伤害、生成或胜负。

敌人推进以 `traversalTimeSeconds` 为唯一权威：

| 类型 | 当前基线 |
| --- | ---: |
| 快速怪 | 14 秒 |
| 普通怪 | 22 秒 |
| 精英怪 | 28 秒 |
| 重甲怪 | 30 秒 |
| Boss | 45 秒 |

道路像素长度、`laneIndex`、`laneOffset` 和视觉 jitter 不改变抵达时间。当前运行时保留 `movementRate`，阶段 2D 可用于减速或加速；本阶段固定为 1。

数值状态：上述时间、准备时长、预告时长、波次间隔和 3～5 分钟目标均为待首关验证、可配置调整。

## 2. 随机与复现模型

| 模型名称 | 设计状态 | 代码状态 | 当前源码位置 | 当前测试位置 | 下一实现阶段 | 需要首关验证 |
| --- | --- | --- | --- | --- | --- | --- |
| run seed 与职责隔离随机流 | 已冻结 | run seed、波次与召唤已实现；combat/merge/skill 仅预留命名流 | `RunSeed.ts`、`SeededRandom.ts`、`RunRandomStreams.ts`、`WaveRandomStreams.ts`、`BattleSession.ts` | `SeededRandom.test.ts`、`RunRandomStreams.test.ts`、`WaveRandomStreams.test.ts`、`HeroBag.test.ts` | 阶段 2D、3、4 接入各自消费者 | 否 |

正式规则：

- 普通新战斗使用 `crypto.getRandomValues()` 生成新的 run seed；安全随机只允许用于生成 run seed。
- URL `?seed=xxx` 优先使用指定 seed，用于调试、复现、分享和自动化测试。
- “同种子重演”保留当前 run seed 并重置全部单局状态。
- “新随机战斗”生成新 seed，不得与同种子重演混淆。
- 玩法随机不得直接使用 `Math.random()` 或直接使用 `crypto`。

独立随机职责：

- `waveCompositionRng`：敌人种类与波次内容。
- `waveLaneRng`：通道、权重与队形分布。
- `waveVisualJitterRng`：局部视觉错位。
- `summonRng`：英雄袋。
- `combatRng`：暴击等战斗随机。
- `mergeRng`：合成、重构及其保护机制。
- `skillRng`：技能候选和刷新。

每波子流的冻结派生格式：

```text
${runSeed}:level:${levelId}:wave:${waveId}:composition
${runSeed}:level:${levelId}:wave:${waveId}:lanes
${runSeed}:level:${levelId}:wave:${waveId}:jitter
```

修改某波数量、通道消费次数或纯视觉 jitter 算法，不得串扰其他用途或后续波次；召唤、战斗、合成和技能也不得互相推进随机序列。

当前 `LevelDefinition.defaultSeed` 只作为直接构造和测试的确定性回退，不是普通玩家新局的固定 seed 策略。

### 3.1 实体移动与渲染同步模型

| 模型名称 | 设计状态 | 代码状态 | 当前源码位置 | 当前测试位置 | 下一实现阶段 | 需要首关验证 |
| --- | --- | --- | --- | --- | --- | --- |
| 权威推进、通道映射与渲染同步 | 架构已冻结 | 敌人固定步长推进、连续通道几何、渲染插值、稳定视图绑定和回退诊断已实现；统一移动效果接口待实现 | `src/battle/PathGeometry.ts`、`src/battle/LanePathGeometry.ts`、`src/battle/BattleSimulation.ts`、`src/battle/RenderInterpolation.ts`、`src/battle/EnemyProgressRegressionMonitor.ts`、`src/ui/BattlefieldView.ts`、`src/ui/state/StableViewRegistry.ts`、`src/ui/state/EnemyViewTransform.ts` | `tests/LanePathGeometry.test.ts`、`tests/RenderInterpolation.test.ts`、`tests/EnemyProgressRegression.test.ts` | 阶段 2D 统一移动效果 | 是 |

权威状态与坐标职责：

- `pathProgress` 是敌人沿共享中心路径推进的权威状态，也是职业攻击区域、默认索敌进度和抵达星核判定的权威数据。
- `laneIndex`、`laneOffset` 和 jitter 只负责敌人的横向分布；它们不得改变推进时间、职业阈值或抵达语义。
- `x/y` 世界坐标由 `LanePathGeometry` 根据 `pathProgress`、通道和 jitter 派生，用于绘制、弹道及后续范围查询，不是战斗推进的权威状态。
- `PathGeometry` 在折点处使用连续顶点切线，`LanePathGeometry` 使用连续法线派生偏移通道；禁止偏移通道在节点处因法线突变发生坐标跳跃。

固定模拟与渲染同步：

- 战斗逻辑继续使用 20ms 固定步长；推进、伤害、索敌、抵达、生成和胜负只在固定模拟步骤中更新。
- 连续移动实体保存 previous/current 模拟进度；每个渲染帧使用 `alpha = clamp(accumulator / fixedStep, 0, 1)` 插值显示进度，再通过 `LanePathGeometry` 派生世界坐标。
- 插值位置只供视觉层消费，不得写回模拟层，也不得决定攻击、碰撞、抵达、状态或胜负。
- 新生成、场景或战斗重置、同种子重演、新随机战斗、后台恢复、丢弃无法追赶的累积时间和视图重建，都必须把 previous/current 与显示基线同步到同一权威状态。

视图身份与写入边界：

- 敌人视图使用稳定 `instanceId` 绑定；`StableViewRegistry` 以实例 ID 为键，不得使用敌人数组下标或排序位置复用视图。
- `EnemyRootContainer`（当前代码中的 `rootContainer`）是运动根节点，其世界 `x/y` 只能由移动渲染器写入。
- `VisualContainer`（当前代码中的 `visualContainer`）承载图形、生命条和表现；受击、缩放、闪白及未来动画只能作用该视觉节点或独立视觉偏移，不能修改运动根节点位置。
- 敌人删除时必须清理视图、Tween、事件和插值缓存；新实例不得继承旧实例的显示状态。

单调推进与未来移动命令：

- 当前没有击退、拉拽、冲锋、传送或换道能力，因此同一存活敌人的模拟进度和渲染进度在正常推进中必须单调不减；阶段 2C.4a 已完成人工浏览器验收。
- 开发回退诊断按稳定实例 ID 区分模拟回退与显示回退，只在超过浮点容差时报告，不得用单调 clamp 隐藏根因。
- 未来只有明确进入统一移动效果管线的击退、拉拽、冲锋、传送或换道命令，才可以改变正常推进规则；每类命令必须定义是否修改 `pathProgress`、lane、速度倍率或仅修改视觉表现。
- 阶段 2D 需要建立统一移动效果接口，至少支持 `slow`、`haste`、`freeze`、`root`、`knockback`、`charge`、`teleport` 和 `changeLane`。其中 slow/haste 调整推进倍率，freeze/root 暂停推进，knockback/charge/teleport 通过显式命令改变进度，changeLane 通过显式命令改变通道；具体叠加、免疫、上限和 Boss 规则待阶段 2D 设计与首关验证。

`CombatStats` 已支持攻击力、攻击速度、暴击率、暴击伤害、技能/对 Boss/召唤物/通用输出倍率、护甲/抗性、物理/元素穿透、承伤倍率、状态强度和控制持续时间。静态定义只提供基础输入，运行时聚合生成不可变快照，不修改 Definition。

属性修改器统一为 `add`、`multiply`、`override`：先基础值加全部 add，再依次应用 multiply，最后选择 override；override 按高 priority 优先、同 priority 按稳定 `sourceId` 排序。聚合结果提供逐属性明细，同一输入集合不依赖创建顺序。

伤害类型：物理、火焰、冰霜、雷电、自然、真实伤害。防御侧支持护甲与元素抗性；进攻侧支持穿甲、元素穿透、增伤和易伤。

暴击基线：

- 大部分英雄默认暴击率 0%；疾风猎手默认 10%。
- 默认暴击伤害 180%。
- 暴击率上限 80%，暴击伤害上限 300%。
- 持续伤害默认不能暴击。
- 所有暴击判定使用 `combatRng`。

阶段 2D.1 已实现的伤害管线：

```text
基础伤害
× 1～4 星伤害倍率
× 通用输出倍率
× 标签对应的技能/召唤倍率和 Boss 目标倍率
× 暴击
× 护甲或抗性减伤（真实伤害跳过）
× 目标 damageTakenMultiplier
= 计算伤害
→ 按剩余生命拆分实际伤害与溢出伤害
```

护甲基线：

```text
减伤率 = 护甲 /（护甲 + 100）
```

减伤上限暂定 70%。百分比穿透先作用于原防御，再减固定穿透，有效防御最低为 0；物理只读取护甲，元素只读取抗性，真实伤害绕过二者。暴击在实际命中结算时使用独立 `combatRng`：非法请求、死亡目标、`canCrit=false`、DoT 或最终暴击率为 0 时不消费；合法判定固定消费一次。`DamageRequest` 先完整校验，`DamageResolver` 纯计算，`DamageApplicationLedger` 在模拟层幂等应用，成功后依次派发伤害与死亡事件。当前默认防御为 0，因此除疾风猎手 10% 基础暴击和正式接入的星级倍率外，原型攻击节奏与基础伤害保持不变。

状态集合包括燃烧、中毒、减速、冻结、眩晕、束缚、破甲、易伤、护盾、治疗、Buff 与 Debuff，仍待阶段 2D.3～2D.4 实现。公式常数、星级倍率、防御值、上限、状态数值和 Boss 控制规则待首关验证。

- `BattleTheme` 选择背景、前景和音频引用；`mistwood-border` 没有正式背景时继续绘制程序雾林，道路与通道几何不变。
- `VfxManager` 统一资源或程序特效的创建与销毁；当前圆点弹道继续使用程序 fallback，VFX 不决定命中或伤害。
- `BattlePresentationEventBridge` 只把已经结算的攻击、死亡和抵达事件翻译为表现指令；动画和音频失败不能改变战斗结果。
- `GameAudioManager` 支持主/音乐/音效/环境音量、静音、暂停、循环去重与场景切换淡入淡出；没有已启用音频时安全静默。
- 资源规格、许可记录和启用流程统一记录在 `docs/asset-pipeline.md`。

当前主题令牌、组件契约、快照驱动 UI、有界提示和清理机制可长期保留；几何英雄、怪物、背景和无音频状态属于临时视觉。

## 16. 多存档模型

| 模型名称 | 设计状态 | 代码状态 | 当前源码位置 | 当前测试位置 | 下一实现阶段 | 需要首关验证 |

冻结规则：

- 3 个独立存档槽；每槽英雄、装备、资源、关卡和挑战完全独立。
- 全局音量、语言和画面设置共享。
- 支持自动保存、重命名、复制、删除、单槽导出/导入和全部存档导出。
- 删除和覆盖必须二次确认。
- 存档包含 `schemaVersion` 并提供版本迁移。
- IndexedDB 保存进度与装备；`localStorage` 只保存全局设置和当前档案 ID。
- 第一版不精确恢复战斗中间状态；未完成战斗可以使用同关卡、同 seed 重新开始。

## 模型变更规则

1. 后续 Codex 任务开始前必须先阅读 `docs/system-models-v1.md`。
2. 修改已冻结模型时，必须明确说明原因，不能以“方便实现”为由静默改义。
3. 变更说明必须列出受影响的代码、测试、存档、配置、UI 和资源契约。
4. 模型变更必须在同一任务中同步更新本文档。
5. 可配置数值可以基于测试和首关体验调整，但不能悄悄改变字段的权威含义或系统职责。
6. 旧规则不得仅因历史文档、测试夹具或原型代码仍存在而重新引入。
7. 当前代码与冻结模型不一致时，先记录迁移阶段和兼容策略，再修改代码；不得把“设计已冻结”直接改写为“代码已完成”。

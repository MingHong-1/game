# 星核守望

《星核守望》是一款原创、单机、无氪金的随机合成塔防游戏。当前仓库已完成战斗基础架构、局内能量、英雄袋召唤、5→10 格位扩展、正式纵向战场、资源接入框架、阶段 2D.1 伤害内核、四名运行时英雄 1 星真实战场图，以及阶段 2.5C.2a 的格位层级与遮挡修复。

## 当前阶段

阶段 2.5C.2a 已完成工程实现：格位底板、HeroEcho/HeroMain、5px 脚底前唇、状态边框、锁层和 HitArea 使用明确显示顺序；英雄不再受整卡覆盖或矩形裁切；选中信息条已移到棋盘顶部。78～80px 尺寸及所有玩法规则保持不变。可用 `?uiDebug=1` 检查各层边界；真实 Canvas 仍待人工复验。下一阶段仍为 2.5D，之后继续 2D.2。

## 技术栈

- HTML5
- TypeScript 7.0.2
- Phaser 4.2.1
- Vite 8.1.4
- Vitest 4.1.10

项目不使用 React、Vue、后端服务、数据库或 CDN 版 Phaser。依赖由 npm 和 `package-lock.json` 管理。

## 环境要求

- Node.js 20.19.0 或更高版本
- npm（随 Node.js 安装）
- macOS 上的现代桌面浏览器

## 安装与启动

```bash
npm install
npm run dev
```

终端会打印本地访问地址，通常为 `http://127.0.0.1:5173/`；如果端口已占用，Vite 会自动选择下一个可用端口。

## 开发命令

```bash
# 启动本地开发服务器
npm run dev

# 执行 TypeScript 严格类型检查
npm run typecheck

# 执行纯战斗逻辑测试
npm test

# 生成生产构建到 dist/
npm run build

# 在本地预览生产构建
npm run preview
```

英雄运行纹理属于离线开发产物，不进入浏览器构建工具链：

```bash
python -m venv .venv-tools
.venv-tools/bin/python -m pip install -r tools/requirements.txt
.venv-tools/bin/python tools/generate_hero_runtime_assets.py
.venv-tools/bin/python tools/verify_hero_runtime_assets.py
```

本地工具虚拟环境不要提交；生成器不会覆盖 1254×1254 母版。

## 当前已实现

- 1280×720、16:9 的固定逻辑画布。
- `FIT` 缩放和双向居中，页面无滚动条，显示区域最大为 1600×900。
- 短暂启动反馈后自动进入主菜单。
- 主菜单展示“星核守望”“随机合成塔防原型”和可点击的“开始新战斗”按钮。
- 原型战场充分使用顶部状态栏与底部指挥台之间的区域；有效中心路径由 `(640,72)` 延伸至 `(640,464)`，实际曲线长度超过 500 像素。
- 明亮星核幻想与雾林边境主题：半透明深蓝面板、金色结构线、青蓝能量光和程序绘制环境。
- 顶部栏集中显示星核生命、波次、时间、敌人数、暂停、倍速、调试开关和返回入口。
- 下方形成三栏星核指挥台：左栏为资源与技能，中栏为居中十格棋盘，右栏为编制、扩格和主要操作。
- 统一 `Panel`、`GameButton`、`ResourceDisplay`、`HeroSlotView`、`HealthBar` 和 `BattleNotice` 组件。
- 单一状态机覆盖准备、战斗中、暂停、技能选择、胜利和失败。
- 固定步长战斗模拟覆盖敌人移动、英雄索敌、自动攻击、弹道、伤害、死亡和星核受击。
- 当前基础攻击已接入无 Phaser 依赖的统一伤害内核；`CombatStats` 支持进攻、防御、穿透和未来状态属性，运行时快照不修改静态 Definition。
- 属性修改器支持确定性的 add、multiply、override；override 按 priority 和稳定 sourceId 决定，当前尚无装备、技能或状态来源。
- 伤害类型支持 physical、fire、ice、lightning、nature、true；当前基础攻击使用 basicAttack/projectile 标签，缺少类型配置时默认为 physical。
- 1～4 星基础攻击倍率为 1/1.65/2.65/4.10；疾风猎手基础暴击率 10%，其他当前英雄为 0%，默认暴击伤害 180%。暴击只由独立 `combatRng` 决定。
- 防御减伤为 `defense/(defense+100)`、上限 70%；百分比穿透先于固定穿透，真实伤害绕过护甲/抗性但仍受输出和承伤倍率影响。
- `damage-applied` 事件包含伤害类型、标签、暴击、实际/溢出伤害和致死信息；表现层只播放反馈，不参与扣血或死亡。开发环境可用 `?combatDebug=1` 查看紧凑结算日志。
- 敌人以 `previous/current pathProgress`、追踪弹道以前后坐标保存固定步长快照；显示层使用限制在 0～1 的累积器 alpha 插值，不改变玩法权威状态。
- 中心路径的顶点切线和通道法线连续插值，外侧通道不会在线段交界处因法线突变而瞬间跳回；`pathProgress`、中心路径和抵达时间保持权威。
- 敌人显示按稳定实例 ID 绑定，数组重排或单位删除不会重分配其他视图；根容器只负责移动，受击 Tween 只作用内部视觉容器。
- 开发模式检测同一敌人的模拟和渲染进度回退；超过 `1e-6` 时只输出一次 `[EnemyProgressRegression]` 结构化日志，不使用单调 clamp 掩盖错误。
- 每帧最多追赶 5 个固定步骤并保留最多 100 毫秒累积模拟时间，超额时间进入开发调试统计；标签页隐藏或窗口失焦后恢复不会追赶后台时间。
- Boss 死亡判定胜利，星核生命归零判定失败。
- 支持暂停、1 倍速和 2 倍速。
- 英雄、敌人、波次、关卡和战斗参数均由独立配置提供。
- 统一 `SeededRandom` 支持字符串或数字种子；URL 可通过 `?seed=自定义种子` 指定初始 run seed。
- 无 URL seed 时，主菜单每次开始新战斗均通过 `crypto.getRandomValues()` 生成新 run seed；局内随机仍只使用 `SeededRandom`。
- run seed 将召唤、战斗、合成和技能随机隔离；波次随机再按关卡 id、波次 id 和用途派生组成、通道、视觉错位三条独立子流。
- 标准关卡使用一条共享中心路径与 5 条隐形横向通道；关卡配置支持 3～7 条通道。
- 走廊顶部、中段、底部宽度分别为 500、430、260；后段五通道仍保留并排防线，不收束为单点。
- 敌人保留统一 `pathProgress`，通道偏移只改变世界坐标；职业攻击区域与格位屏幕距离均不受通道影响。
- 敌人推进以 `traversalTimeSeconds` 为唯一权威：快速斥候 14 秒、普通爬行者 22 秒、重甲 30 秒、精英 28 秒、Boss 45 秒；更换道路像素长度不会改变抵达时间。
- 原型关卡配置 8 秒开局准备期，准备期间允许召唤且不生成敌人；每波开始前 5 秒提供敌人、威胁类型和队形预告。
- 波次在准备期后的 0、18、45 秒启动，保留时间重叠和怪潮压力；完整关卡 3～5 分钟目标仍需第一关垂直切片验证。
- 开发调试面板显示最近敌人的配置推进时间、预计剩余时间、首次实际抵达耗时、平均推进时间、同屏峰值和当前波次开始时间。
- 波次支持随机、均匀、中央突袭、两翼、全线怪潮和 Boss 中央/小怪两翼六种通道模式。
- 每波的敌人组成、通道分配与局部横向错位分别由 `waveCompositionRng`、`waveLaneRng`、`waveVisualJitterRng` 可复现地产生；三者互不推进，也不影响召唤随机流。
- 波次子流种子格式为 `${runSeed}:level:${levelId}:wave:${waveId}:composition|lanes|jitter`；修改前一波数量或纯视觉 jitter 消费不会串扰其他波次或玩法内容。
- 当前原型初始能量 15；普通、重甲、精英击杀分别奖励 1、2、5，波次完成奖励由关卡配置提供。正式经济已冻结为阶段 2E 的十位数波次预算模型。
- 当前原型召唤费用采用 5、6、7、8、9 五档；阶段 2E 在不改变袋和格位架构的前提下迁移为 50、60、70、80、90。
- 召唤袋每轮包含 4 名英雄各 2 份，抽完后重新洗牌生成下一袋。
- 初始 5 格，在第 5、10、15、20、25 次成功召唤后扩展到 6、7、8、9、10 格。
- 十格始终可见：下排第 1～5 格开局开放，上排第 6～10 格从左到右依次解锁；只有下一个锁定格显示详细进度。
- 满格时通常在抽袋前零副作用拒绝；若下一次成功召唤将扩格，则允许先扩格再原子完成扣费、抽袋、占位和计数。能量不足仍不会解锁、扣费、抽袋或推进次数。
- 胜负面板明确区分“同种子重演”和“新随机战斗”。
- 当前四名运行时英雄以真实 1 星运行纹理固定显示在所属格位；格位默认只显示主体和必要状态，点击后在统一信息条查看名称、精确星级、职业、攻击与暴击率。图片不可用时安全回退几何图形，具体格位不提供攻击、范围或目标选择加成。
- 局内 `HeroStar` 合法范围为 1～4，等价值为 1/2/4/8；4 星显示为满星并继续保留攻击能力，非法范围在运行时边界明确失败。
- 英雄从格位中的可见位置发射弹道，目标合法性只读取敌人的 `pathProgress`：射手、法师、辅助、战士分别从 0.05、0.20、0.35、0.70 开始普通攻击，格位屏幕坐标不参与射程判断。
- 右侧抽屉默认收起；正式提供真实波次/怪物情报和按英雄汇总的本局伤害统计。开发环境或显式调试参数增加 FPS、模拟步数、插值 alpha、丢弃追帧时间、敌人/弹道/Tween、种子和状态等模拟调试标签。
- Canvas 逻辑缓冲保持 1280×720，未额外乘以设备像素比；配合 `FIT` 与 1600×900 页面上限，避免 5K 屏幕创建无上限渲染缓冲。
- 支持能量差值、召唤/扩格、敌人受击、星核受伤、波次、Boss 和胜负的轻量视觉反馈。
- 显式资源清单只将 `enabled=true` 的本地资源加入 Boot 加载队列；重复 ID/key、空路径和远程路径在开发阶段被拒绝。
- 五张 1254×1254 RGBA 文件作为高分辨率母版保留；离线工具生成五张 256×256 RGBA 运行纹理，当前清单只启用四名运行时英雄的运行纹理。森灵唤师、3/4 星、头像、怪物、背景、音频和特效均不请求。
- 四名英雄使用 78～80px 整数目标显示尺寸、84px 上限；脚底 anchor 和整数 slot offset 分别由视觉定义配置，程序 fallback 尺寸不变。最终遮挡、武器边缘和清晰度仍需在真实 Canvas 复查。
- 1～4 星拥有纯表现组合：2/3 星使用无实例语义的视觉分身，4 星使用单主视觉和满星光环；这些节点不攻击、不参与输入或统计。
- 格位已预留统一 hovered/selected/dragging/mergeEligible 等交互样式和矩形 HitArea，但没有真实拖拽、合成意图、`MergeSystem` 或 `mergeRng` 消费。
- 局内英雄统一为 1～4 星；视觉按 1/3/4 星节点选择并向低阶回退，怪物按实际 `enemyId` 选择图片、Sprite Sheet 或程序 fallback。
- `HeroBattleView`、`EnemyBattleView`、动画状态、`BattleTheme`、`VfxManager` 和表现事件桥均与战斗规则分离。
- `GameAudioManager` 支持分类音量、静音、循环去重、暂停恢复和淡入淡出；当前无音频资源时安全静默。
- 战场、怪物、星核和 UI 仍使用程序视觉；四名运行时英雄已使用本地 PNG，不使用远程资源。
- 资源规格、许可和首个正式资源接入步骤见 `docs/asset-pipeline.md`。
